# Streaming destination ingress

## Purpose and scope

This specification governs pre-finalization destination staging, final package binding, capability selection, crash behavior, evidence, and memory/backpressure integration. Ordinary receipts, idempotency, package identity, and checkpoint gating remain governed by their existing specifications.

## Behavioral contract

Only a durable package segment whose bytes and SHA-256 are complete MAY enter staged ingress. The runtime MUST preserve deterministic manifest order independently of arrival/scheduling order.

A staged-ingress request MUST carry a `LoadAttemptId`, destination/target identity, planned disposition/schema/plan authority, the complete compiled output Arrow schema, disposition inputs such as merge keys, and bounded scheduling context. It MUST NOT claim a package hash or package-token idempotency guarantee. Driver-specific schema preparation remains inside the destination adapter; the generic runtime MUST NOT branch on destination identity or physical provenance encoding.

The selected `PreparedBulkPath` is the plan-time physical policy authority. It MUST fix the driver path/version, writer count, batch bounds, ordering, and versioned object/key derivation policy before staging mutation begins. Values that cannot exist before package finalization, including the package-hash idempotency token and final keys derived from it, MUST be produced only by applying that recorded policy to the verified final binding; they MUST NOT cause runtime replanning. Attempt metadata MUST record the selected physical policy and exact writer/batch settings so recovery and evidence can distinguish incompatible attempts.

A staged-segment request MUST carry segment id, SHA-256, row/byte counts, schema hash, ordinal, and a bounded Arrow batch stream or segment reader. The reader retains the source memory accounting until the destination finishes or transfers it into equally authoritative destination accounting. A destination MAY hold multiple unacknowledged requests only within the request's declared segment/byte scheduling bounds. It MUST acknowledge each exact identity accepted into its declared staging scope or fail; completion and acknowledgement MAY arrive out of order, but snapshots and final binding MUST restore canonical ordinal order. Partial acknowledgement is forbidden. A versioned `PreparedBulkPath` MAY deterministically group consecutive canonical package segments into one destination object while preserving every segment's row offset, row count, byte counts, acknowledgement, and provenance. Group boundaries MUST depend only on recorded plan policy and canonical segment metadata, never scheduling or completion order. The plan MUST record the object-key derivation policy; a value-dependent immutable key MAY be instantiated from hash-while-write evidence after encoding without replanning. Such content-addressed objects MUST remain invisible until a verified manifest binds them, MUST be create-or-verify, and MUST use independent reachability authority for reclamation. The capability MUST declare `resumable` or `rollback_redrive` recovery and MUST NOT imply persistence across process loss for ephemeral transactions. A `rollback_redrive` adapter MAY acknowledge a fully consumed source segment with `external_durable = false` before its containing destination object is durable only when the verified package remains the durable replay authority and abort/retry deterministically redrives the entire group.

Before final binding, staged data MUST be invisible to target reads that represent committed destination state. No package receipt, `_cdf_loads` committed row, final object manifest/pointer, checkpoint, or delivery guarantee may be emitted.

Final binding MUST occur only after local package finalization and verification. It MUST provide the verified package hash/token and complete ordered segment identities. Any missing, duplicate, extra, reordered-without-declared-order-independence, or mismatched staged identity MUST fail before target publication.

Staged writes and final binding MUST execute through the blocking lanes declared by destination capabilities. The generic host owns scheduling and admission; adapters own native connection and transaction behavior.

Successful final binding MUST produce the existing receipt shape over the final package hash and committed segment acknowledgements. When final binding itself has exactly verified every published mutation from hash-while-write evidence, create-only publication, durability barriers, and the completed manifest, it MAY return commit-bound receipt verification for that exact receipt id. The generic gate MUST still validate receipt/state structure and exact receipt-id binding before accepting that evidence. Duplicate, recovered, or otherwise pre-existing commits MUST use independent runtime verification. This is an optimization of when verification occurs, never permission to omit it; checkpoint commit still follows only verified receipt evidence.

For object-store replace semantics, immutable per-package settlement evidence MUST be distinct from the mutable target-current pointer. Data objects, provenance, the package manifest commit marker, and per-package settlement MUST publish create-only. The mutable pointer MUST advance through one provider-atomic compare-and-swap authority with a strictly increasing generation; a successful writer MUST read back its exact pointer bytes before returning commit-bound verification. Exclusive-writer serialization is not required when that CAS authority is available, and a provider that cannot supply it MUST fail the capability preflight. Settlement MUST publish only after the current-pointer CAS succeeds, so its existence proves the package completed replace publication; duplicate receipt verification MUST use that immutable settlement and MUST NOT repoint a target that has since advanced. A missing settlement after a published package marker MUST be idempotently repairable by re-entering the CAS protocol; it MUST NOT overwrite immutable evidence or use wall-clock order.

Abort MUST be idempotent. Every externally durable staging attempt MUST execute under one runtime-owned `StagingLease`; destination adapters MUST NOT create their own heartbeat, liveness thread, or retention eligibility policy. The lease identity binds destination, target, and load attempt; acquisition assigns a monotonically increasing fencing token inside a stable lease-authority domain. A staged namespace MUST include that domain as well as the token, so two independent authorities cannot alias the same attempt namespace. The runtime or artifact/storage authority MUST acquire and renew the lease independently of segment completion, using an injected `ExecutionHost` task scope rather than destination- or runtime-private threads. It MUST retain the lease through final binding or abort and release it only after all adapter work is structurally joined.

Every externally durable mutation MUST carry a runtime-issued `StagingMutationGuard` for the exact lease generation. The adapter MUST assert that guard immediately before and after each externally visible mutation and at bounded batch or upload-part boundaries during long operations. Renewal failure MUST cancel the guard and stop further publication or acknowledgement; destination-specific cancellation mechanisms MAY shorten that bounded response time but MUST NOT replace the generic guard. Abort and finalization MUST preserve and report cleanup, worker-join, and lease-release failures alongside the primary failure.

Every externally staged namespace and cleanup candidate MUST record the lease-authority domain, lease identity, and fencing token that created it. A restarted attempt MUST use a fencing-token-distinct namespace, so cleanup of an older generation cannot delete newer work. Successful binding and explicit abort MUST remove the exact attempt generation immediately. Process-loss cleanup MUST atomically advance the fencing generation into a renewable cleanup lease as part of the authoritative `prove_expired` operation; a passive proof followed by deletion is forbidden because another writer could acquire between those steps. Object age, modification time, process-local ownership, or completed-segment activity is never sufficient evidence. The collector MUST hold the cleanup lease and its exact mutation guard until deletion finishes, MUST delete only namespaces exclusively owned by the exact proven-expired generation, and MUST report storage errors rather than claim cleanup. A cleanup commit marker MUST be deleted last: if any payload deletion fails, the marker remains enumerable so the same fenced cleanup generation can resume. Shared content-addressed objects that a later generation can reuse are not staging-owned and MUST NOT be deleted by this path; their reclamation requires independent reachability authority. Recovery MAY reattach only while holding the matching live lease and after destination-reported staging identities exactly match the run ledger and final package where available.

All staging buffers, queues, transactions, temporary files, atomic-upload payloads, and recoverable multipart parts under CDF control MUST register with the P3 memory ledger or declare external durable storage. A process-local provider multipart handle is not recoverable staging authority and MUST NOT back an externally durable claim. Attempt-owned durable bytes MUST stay within a declared bound independent of total run size. Finite immutable content-addressed objects are not attempt staging after create-or-verify publication; they require reachability GC and MUST NOT be copied again during final binding. Backpressure propagates from staged ingress to package persistence and upstream operators through byte permits.

## Capability and conformance

Destination inspection/sheets MUST declare finalized-only versus staged ingress, visibility isolation, rollback/cleanup support, writer concurrency, and useful in-flight byte/segment bounds. Planner/runtime selection MUST be capability-driven.

Conformance MUST cover successful overlap, bounded multi-segment ownership, out-of-order acknowledgement with canonical final order, staged-write failure, renewal failure during a long mutation, crash after every staged segment, resumable reattach, rollback/redrive, abort repetition and error preservation, final-binding mismatch, duplicate finalized package, no receipt before binding, no checkpoint before verified receipt, cross-authority namespace isolation, marker-last resumable staging cleanup, and finalized-only fallback. Jobs 1/N MUST yield identical package hashes and receipt package/segment identities; destination transaction metadata may truthfully differ.

## Explicit exclusions

This spec does not prescribe one lease persistence backend, make staging a package artifact, change SHA-256, permit direct unverified source-to-destination writes, or require every destination to overlap ingestion. Standalone execution uses the configured durable state authority; distributed runtimes MUST inject a lease authority whose consistency domain covers every worker and collector that can touch the staged namespace.
