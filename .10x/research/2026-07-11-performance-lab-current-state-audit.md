# P3 performance lab current-state audit

## Question

What does `cdf-benchmarks` measure today, what would make its results misleading at P3 scale, and what exact seams should L1-L5 preserve or replace before recording the baseline?

## Sources and methods

Read the P3 envelope decision/spec and L1-L5 tickets; inspected every `cdf-benchmarks` source, fixture, test, script, dependency, Criterion case, and trend record; traced benchmarked public runtime paths; inspected the run-event schema/ledger emission and searched for existing memory/RSS/spill accounting.

## Findings

### Foundation worth preserving

The crate is private and opt-in, fixture specs are committed while generated bytes are not, fixture generation is deterministic, format/package/destination cases call public CDF APIs, metric labels distinguish trend versus ad-hoc comparisons, and Postgres is isolated behind an explicit environment opt-in. The existing matrix is a useful smoke/correctness seed.

### Current measurements are not P3 evidence

The committed fixtures are only 8 rows, 2,048 rows, and 512 rows/64 columns. Fixture generation materializes complete Arrow batch vectors and complete CSV/JSON/NDJSON byte vectors, so it cannot generate or validate the 100 GB constant-memory law.

`run_case` constructs fixtures/packages/destinations during the measured operation. This is appropriate for a named end-to-end/startup case but invalid for decoder/package rooflines unless setup and timed regions become explicit per workload. Native and CDF cases do not currently perform equivalent setup or semantic work.

Criterion is used for every case with ten two-second samples. It is suitable for microkernels but does not define macro process isolation, cache state, timeout, CPU time, peak RSS, spill, phase metrics, or external reference execution. The trend binary runs each case exactly once and appends a flat record containing timestamp, label, elapsed time, rows, and one ambiguous byte count.

The trend schema has no version, fixture identity, CDF commit, dependency/reference version, host fingerprint, warm/cold mode, sample distribution, failure/unavailable status, logical versus physical bytes, CPU time, peak RSS, or bias label. Appending different machines and revisions to one path creates incomparable records without detection.

Several byte metrics are output Arrow memory size rather than bytes read/written; native DuckDB reports zero. Therefore the current numbers cannot produce meaningful throughput or roofline ratios.

The event spine carries useful row/byte facts at package/destination boundaries and one run-failure elapsed value, but does not carry balanced decode/validate/encode/persist/hash/destination/finalize/gate timing. No general runtime memory ledger, peak RSS, or spill authority exists yet, as expected before P3 WS-A/F.

The benchmark runner imports and constructs concrete destination helpers from `cdf-project`; the P0 destination-boundary migration must leave the lab consuming the same explicit registry composition contract rather than creating benchmark-only adapters.

The benchmark README still states there are no CI hard gates and labels Polars deferred. Both are historical P0 posture that L3/L4 must supersede; Polars should remain externally isolated so it does not contaminate the Cargo graph.

## Conclusions

L1 must define separate dataset recipes, workload definitions, timed-region/setup policies, versioned observations, and host/reference identities. Large generators must stream bounded chunks directly to sinks or expose a generated resource; returning complete byte vectors is prohibited for scale fixtures.

L3 must use a process-isolated macro runner for wall/CPU/RSS/timeout/cache-state measurement and reserve Criterion for microkernels. Each case must explicitly name setup included/excluded and logical/physical byte counters. Raw samples are retained; reports derive medians and dispersion rather than appending isolated points.

L2 must add generic phase telemetry to the existing event authority, not a benchmark side channel. L4 compares only a full comparability key and cannot treat unavailable/inconclusive cells as green. L5 should record the current materialization failures honestly; it is not required to make a 100 GB run succeed before the optimization program that exists to enable it.

The user's latest priority supersedes P3's stale dependency on complete P1 closure. WS-L may execute immediately; P1 WS8/WS9 may proceed without changing the pre-optimization data plane, and later P3 runtime work still waits for L5.

## Limits

This audit did not run timing cases because current outputs cannot support the P3 claims and running them would create an unlabeled pseudo-baseline. It does not choose an RSS backend or cache-eviction command; L3 must implement portable capability detection and record unsupported modes honestly.
