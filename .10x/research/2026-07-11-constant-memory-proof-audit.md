# Constant-memory proof audit

## Question

What evidence and enforcement prove that a 100 GB/1 TB run stays under a configured process budget across CDF, native libraries, metadata, filesystem I/O, and child adapters rather than merely balancing one internal pool?

## Sources and methods

Inspected active memory/envelope decisions/specs, P3 runtime/codec/destination/package/remote tickets, production collection/buffering patterns, benchmark host capability records, and current process/RSS tooling availability. Classified buffers as managed, calibrated native/runtime headroom, external durable staging, OS/file-backed, or child-process memory.

## Findings

The active decision correctly defines the user budget as a process RSS ceiling and a smaller shared managed pool. Exact headroom remains intentionally blocked on L5. The current implementation has no finite pool or process enforcement yet.

Confirmed package/input-sized production materializations include format `FormatRead.batches`, full compressed/text bytes, JSON DOM/pages, engine pending dedup batches/keys/provenance, package reader commit vectors, destination staged segments/row vectors/full Parquet batches, transport listing/response vectors, and manifest/evidence metadata collections. Focused P3 owners now exist for each class.

An internal ledger can be perfectly balanced while RSS exceeds budget due to allocator fragmentation/caches, Arrow/native library scratch, DataFusion execution, compression/parser internals, database clients, mmap/file mappings, thread stacks, runtime queues, or Python/subprocess children. Therefore RSS/process-tree measurement independently falsifies the ledger and calibrates headroom.

Linux provides the enforceable slow-tier host through cgroup v2 aggregate memory plus process high-water counters; macOS can measure with `time -l`/host sampling but cannot provide identical cgroup enforcement. Cgroup `memory.current/peak` includes page cache, so reports must distinguish aggregate cgroup memory, process-tree RSS/high-water, and managed-pool peak rather than comparing unlike values.

The stress generator itself must be constant-memory and outside the measured child process or it can invalidate the proof. Input files/streams are generated deterministically in bounded chunks; generated bytes are not committed. A 100 GB test cannot rely on compressing one giant in-memory value.

Children/foreign runtimes require aggregate authority. In-process native allocations consume calibrated headroom and telemetry. Subprocess adapters must run inside the same cgroup/job/process-tree budget or declare an enforceable child sub-budget; otherwise the standalone product cannot claim a process-tree ceiling.

## Conclusion

Define a process-tree proof protocol with three simultaneous observations: resolved process budget/cgroup authority, shared managed-pool accounting, and OS high-water/aggregate memory. Linux slow CI enforces the 2 GiB limit; portable hosts record comparable-but-not-enforced observations.

Exercise volume, cardinality, compression, skew, quarantine, dedup spill, remote readahead, destination staging, and foreign children. Require completion and clean below-minimum failure. Reconcile every materialization owner and forbid unclassified production buffers at WS-F closure.

## Limits

Exact default headroom, cgroup margin/file-cache treatment, allocator choice, sampling cadence, and stress duration are selected by L5/F1 calibration. This audit does not pre-approve an allocator dependency.
