## Review

### Findings

- **Significant — the fourth-driver law is inspection-only, not the required generic runtime/conformance proof.** `fourth_registered_destination_inherits_the_generic_bulk_matrix_contract` registers `FourthDriver`, calls only `DestinationRegistry::inspect`, and applies shallow assertions over nonempty descriptor/evidence strings, one selected path, nonempty dispositions, and non-`None` idempotency. `FourthDriver::resolve` deliberately returns `"fourth matrix fixture is inspection-only"`. The fixture therefore cannot lock, plan, resolve, ingest, settle, replay, resume, or doctor, and it cannot exercise schema eligibility/rejection, forced fallback, staged/finalized ingress, abort/redrive, duplicate package, receipt verification, crash boundaries, memory pressure, dispositions, or jobs invariance. This does not satisfy this ticket's “full generic matrix” scope or the acceptance scenario in `.10x/specs/destination-extension-runtime-contract.md`. The observed 4/4 conformance result proves only catalog inspection/static import assertions within its stated limits.
- **Significant — the published green envelope is manually claimed data, not registry/machine-report-derived measured evidence, and it omits required cells/host provenance.** `DestinationPathEnvelope` stores destination, path id, evidence version, observation, status, and evidence link as free-form strings in `p3-envelope-spec.json`; validation checks only nonempty strings and duplicate `(destination, path_id)` pairs. Generation never joins these rows to `DestinationRuntimeCapabilities`, the benchmark report, the cited evidence record, or a host/comparability key. Consequently a path id/evidence version/rate/status can drift or be invented while the envelope golden and the 4/4 catalog tests remain green. The three rows also inherit the unrelated pre-optimization report host header even though their D2-D4 measurements are separate observations, and neither the rows nor their leaf evidence identify the named host class required by this ticket and `.10x/specs/performance-lab-and-envelope.md`. Finally, the matrix publishes only three selected green paths: it has no explicit schema-ineligible/unavailable cells, despite this ticket requiring unavailable/ineligible cells to remain visible. The leaf D2-D4 records remain valid for exactly their stated local adapter observations; they do not authorize presenting this second catalog as a host-labeled generated measurement matrix.

### Verdict

**Fail.** No implementation defect was found in the one-line DuckDB ingress correction, and the scoped diff does not resurrect scalar DuckDB or CSV/text PostgreSQL production fallbacks or add a concrete-destination branch to shared runtime. However, both D5 proof obligations above remain unsatisfied. Pending CLI/envelope tests cannot close them because the assertions currently encode the same shallow/manual authorities.

### Residual risk

- CLI execution remains unverified because the two completed doctor assertions came from interrupted commands and were explicitly recorded as non-evidence. Static inspection shows doctor renders missing descriptors, undeclared selection, and absent evidence strings, but it does not establish that the evidence version corresponds to a measurement or that the global and selected-path evidence versions agree.
- The DuckDB descriptor now truthfully matches its live `StagedDurableSegments` runtime mode; PostgreSQL remains binary-COPY/finalized-only and Parquet remains streaming-segment/finalized-only in the inspected descriptors. This review did not rerun D2-D4 adapter suites and relies on their accepted closeout evidence for binary framing, Arrow ingestion, bounded Parquet writing, idempotency, disposition, and receipt behavior.
- Focused CLI/envelope tests remain pending a stable J3 source as journaled. No build or test was run during this review.

## Retrospective

- The stale DuckDB descriptor said finalized-only even though its live runtime is staged durable ingress. Joining descriptor and live capabilities in one catalog law found this drift without a destination branch.
- “Forced fallback” did not authorize reviving old implementations. The current-format decision and D2/D3 closeouts establish scalar DuckDB and text/CSV PostgreSQL as removed benchmark controls; production has no truthful second path for those envelope schemas.
- The benchmark lab's original machine report remains an intentionally red pre-optimization baseline. Replacing its historical observations would destroy evidence. A focused generated destination-path section preserves that baseline while consolidating current D2-D4 closeout authority.
- The first conformance build was expensive because concurrent J3 source changes invalidated shared dependencies. The narrow law itself completed in 0.05s after compilation; subsequent validation must remain slot-coordinated rather than broad.
- The unused bulk-writer hierarchy survived because descriptor design and writer execution were reviewed separately. Searching for production implementors/callers before closure exposed the split. Future extension contracts must prove that declared preparation data reaches the actual mutation trait, not merely that both types exist.
- The Parquet roofline regression was useful: absolute encode throughput improved while the device control improved more, so the ratio caught syscall amplification that a fixed absolute target would have missed. Hashing, durability, and spill accounting remained in the measured path.
