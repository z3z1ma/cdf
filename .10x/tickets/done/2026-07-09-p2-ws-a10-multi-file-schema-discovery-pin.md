# P2 WS-A10 multi-file schema discovery and pinned manifest identity

## Scope

Parent plan: replace single-file discovery slices with a format-neutral resource-level compiler stage and integrate its aggregate schema, manifest, effective-schema, per-file verdict, quarantine, and checkpoint evidence through the ordinary pipeline. A10a/A10b establish parallel artifact and pure-join foundations; A10c integrates exhaustive local binary discovery; A10d adds compatible runtime evidence; A10e adds gate-backed file quarantine; A10f owns conformance closure.

## Acceptance criteria

- `cdf schema discover|pin`, `plan --no-pin`, and first-use auto-pin accept multi-file Parquet and Arrow IPC file resources and never require narrowing a glob to one file.
- Exhaustive discovery probes every footer/schema block without reading data pages or computing runtime full-file hashes. Explicit `sample_files = N` uses `stratified-hash-v1`; resolved per-executor budgets are evidence and exhaustion never activates or changes sampling.
- Default `evolve` produces one deterministic union schema using only ratified lossless widenings; missing fields, new fields, incompatible physical types, and metadata variance have explicit recorded semantics.
- First exhaustive pin requires the exhaustive compatible aggregate; first sampled pin requires the selected compatible aggregate and records every unprobed entry honestly. Existing baseline pins stay immutable; `evolve` derives a recorded effective schema and `freeze` quarantines deviations.
- The pinned artifact identity includes aggregate schema, normalizer/policy versions, coverage/selector evidence, and a sorted discovery manifest with content identity and participation for every matched file; only probed files carry physical-schema hashes and discovery verdicts.
- Adding/removing a file or changing its bounded observation identity changes the fresh discovery manifest even when the aggregate Arrow schema is unchanged; runtime exact identity remains authoritative for data-only changes outside bounded discovery evidence. Ordinary commands never refresh the baseline pin implicitly.
- Plan/preview/run share the pinned aggregate schema and per-file reconciliation front end. Runtime exact `FileManifest` SHA/ETag evidence and append incrementality remain intact.
- Deterministic local multi-file Parquet/Arrow IPC and transport-fixture tests cover compatible union/widening, missing fields, incompatible quarantine, exhaustive and sampled coverage, unseen runtime residual/quarantine routing, no-change identity, changed-set diff, bounded large-N behavior, and no-write failures.

## Evidence expectations

Product-path byte-count evidence, multi-file schema fixtures, deterministic discovery-manifest snapshots, per-file verdict/package evidence, pin/diff/autopin tests, runtime manifest regression coverage, full affected-crate tests, and adversarial review.

## Explicit exclusions

This ticket does not add Arrow IPC stream framing, Parquet/IPC row decoding changes, compression support, runtime manifest identity changes, cloud credential payload syntax, HTTP template enumeration, text sampling, or destination mapping policy. Remote facade composition is required, but new transports remain WS-E.

- 2026-07-09: Read-only architecture audit completed after the ticket was reshaped; no behavior or tests were changed. The current path is:
  - File declarations compile in `crates/cdf-declarative/src/compiled.rs`: `FileResourcePlan` owns one resolved `format`, `format_declared`, root, and glob. D5's `local_file_discovery_candidates` in `file_runtime.rs` is the reusable bounded local selector: deterministic contained paths plus stat/compression/extension/magic evidence without the runtime SHA-256. It is not yet transport-neutral and exposes neither modified time nor a typed content identity.
  - Runtime planning is deliberately stronger and must remain separate. `resolve_file_matches`/`resolved_file_match` compute the exact local SHA-256, `partition_for_file_match` records it in every `PartitionPlan`, `validate_partition` re-resolves/rechecks it, and `cdf-project::runtime::orchestration::file_positions_from_partitions` drives append incrementality from those exact identities. `file_source_position` in `cdf-formats/src/readers.rs` hashes again when a local file is opened. A discovery aggregator must not call these runtime paths, but must preserve them unchanged for `FileManifest` evidence.
  - The transport seam is `FileTransport::{metadata,read_range,list}` plus `FileRuntimeDependencies::{with_transport,range_reader}`. It is reusable for candidate probing, but its current local `list` is non-recursive and calls `local_metadata`, which hashes every file; HTTP `list` intentionally rejects arbitrary listing; `FileTransportLocation` has only local/file/HTTP variants. A generic discovery candidate API therefore has to sit above this trait or split bounded discovery metadata from exact runtime metadata. It cannot reuse `list` as-is without recreating the full-read regression.
  - The bounded probe adapters already exist in `cdf-formats`: Parquet has local/footer and generic `ChunkReader` probes plus `RangeChunkReader`; Arrow IPC file framing has a `Read + Seek` schema-block probe and measured local byte reads. Parquet's local probe does not currently expose measured bytes, and Arrow IPC has no range-backed remote `Read + Seek` adapter. Both return one physical Arrow schema and per-file evidence suitable for adapters into a common discovery result.
  - `cdf-project/src/schema_discovery.rs` is the present dispatcher and aggregation boundary, but it normalizes and snapshots one schema immediately. `single_local_parquet_partition`, `single_local_arrow_ipc_candidate`, and `single_http_parquet_partition` are the explicit cardinality gates. The Parquet local gate calls runtime `plan_partitions`, causing the full-hash regression; Arrow IPC uses the bounded candidate selector; HTTP Parquet uses the runtime single-URL planner. Remote Arrow IPC is rejected before probing. `LocalParquetSchemaDiscovery` also embeds exactly one partition, a format-shaped result that should disappear behind a resource-level discovery set.
  - Snapshot authority is split between an ephemeral `DiscoveredSchemaSnapshot.source_identity` map and persisted `SchemaSnapshotArtifact`. The map is rendered by `schema discover|pin` but is not hashed or stored. Artifact version 1 hashes only `(version, resource_id, aggregate schema, string metadata)`. `SchemaSnapshotReference` carries only hash/path/string metadata into `cdf.lock`; ordinary `plan|explain|preview|run` hydrate the Arrow schema from the verified artifact in `cdf-cli/src/context.rs` and do not contact the source. Explicit `schema discover|pin|diff` and `plan|explain --no-pin` make a fresh probe. This authority ordering is reusable and must not regress.
  - `schema pin` and `prepare_discover_resource_for_cli` can reuse one aggregate result with no conceptual CLI fork, but their reports only model one flat source-identity map. `SchemaDiffReport` compares fields and string snapshot metadata and does not treat a hash-only discovery-set change as drift. The CLI needs typed manifest summary/diff reporting (matched/probed/admitted/quarantined/unprobed plus added/removed/changed files) while retaining no-write guarantees.
  - At execution, Parquet and Arrow IPC readers independently reconcile each physical file against the pinned schema through `cdf-contract::reconcile_schema`, materialize casts, and attach one `SchemaCoercionPlan` to each batch. This is a useful per-file front end, but missing pinned fields are currently fatal, extra physical fields are projected away as `Extra`, and no null array is synthesized for union fields absent from one file. `cdf-engine::execute_to_package_inner` then requires every batch's coercion plan to be identical and emits one `schema/coercion-plan.json`; a legitimate multi-file `widened` versus `preserved` pair therefore fails even when both materialize the same aggregate schema. A10 needs resource-level plus per-file coercion evidence, not relaxation of evidence validation.
  - Existing quarantine is row-oriented. Readers can attach `PreContractQuarantineFact`, and the engine writes quarantine Parquet artifacts, but a schema reconciliation error occurs before a batch exists. There is no file-level verdict carrier. More importantly, checkpoint output positions are derived only from accepted output segments; a quarantined file with zero accepted rows yields no segment position, and an all-quarantine run fails with `package execution produced no output segments to checkpoint`. A file-quarantine contract must explicitly decide whether quarantine advances the runtime manifest, then add processed-file position evidence independent of output segments if it does.
  - The plan/package spine currently preserves the pinned snapshot hash as the destination/state schema hash and serializes scan, explain, validation-program, output schema, and coercion artifacts into package identity. It does not copy the pinned snapshot or discovery manifest into the package, and `EnginePlan` has no snapshot/discovery-manifest field. To satisfy the standing snapshot-stamped-into-plan/package rule, A10 needs a reference or canonical discovery-manifest artifact in the plan/package identity in addition to the state schema hash.

- 2026-07-09: Evaluation of the proposed 4,096-file/64-MiB + first/last/hash-ranked contract found additional execution-critical ambiguity; the current checkpoint is not sufficient to implement safely:
  - "Hash-ranked" does not name its hash input. Full content hashes violate bounded discovery; path hashes are bounded but do not rank by content identity. The ranking key, tie-breaking, first/last counts, remaining sample count, and whether either the file-count or byte limit stops probing must be exact.
  - Sampling conflicts with the requirement that every matched entry carry a physical-schema hash and verdict. An unprobed file has neither. The contract must either (a) probe every binary footer/schema block and treat the byte budget as a hard diagnostic/failure requiring override, or (b) permit explicit `unprobed` entries and state that the pinned aggregate is sampled, not exhaustive. It must then define whether first-use auto-pin/run is allowed from a sampled aggregate and how unprobed runtime drift is reported.
  - "Content identity" is underspecified for bounded local discovery. Size + mtime + footer/schema hash does not detect data-page-only changes when those values are preserved; exact SHA-256 requires the forbidden full read. The contract must name a transport-specific bounded discovery identity and its limitation, or authorize exact hashing. Stable/multipart ETag semantics also need classification rather than treating every ETag as content identity.
  - The aggregate join needs a closed least-upper-bound table, not only directional widening examples: signed/unsigned combinations, int/float, decimal precision/scale, timestamp units/timezones, nested children, dictionary/view/run-end types, nullability, schema/field metadata, and normalizer collisions all need deterministic outcomes. "Sorted first appearance" must define file order, source-field identity, and field order. Missing fields need a `cdf:null_origin`/physical-provenance rule and runtime null materialization.
  - Metadata conflict policy is wholly unresolved. Source names and CDF reserved metadata cannot be silently dropped; arbitrary schema/field metadata may conflict across files. The contract must classify merge, preserve-per-file-only, reject, and reserved-key behavior.
  - `freeze` first baseline is circular as "first sorted conforming file": conforming requires a baseline. The exact first candidate eligibility, malformed-file handling, zero-conforming behavior, and whether nullable/schema metadata come from only the baseline must be stated. Refresh of an existing freeze pin also needs the existing artifact passed explicitly as authority; current fresh-probe APIs discard that context.
  - Incompatible-file disposition must distinguish discovery-time membership from run-time processing. Define whether incompatible files are excluded, quarantined as file units, or fatal under each policy/trust preset; the stable rule/error code and remediation; whether their identity participates in the pin; whether quarantined files advance append `FileManifest`; and whether a corrected file with the same path is retried.
  - The authority for `evolve` versus `freeze` must be named. Today execution derives `ContractPolicy` from `trust_level`; a custom `ContractRef` is not loaded into discovery. A10 needs either the trust preset as explicit authority for this slice or a concrete contract-resolution input, plus a version/hash recorded in the discovery artifact.
  - Per-file physical provenance and coercion cannot fit the current singular `cdf:physical_type` field metadata and singular `ValidationProgram.schema_coercion`. The contract must authorize a canonical per-file evidence model while keeping the aggregate field's output type authoritative.
  - Removal semantics need confirmation: a fresh pin omitting a formerly matched file changes discovery identity, but runtime append state normally retains prior manifest entries. State whether schema refresh/removal affects only future schema authority or also requests data deletion/replacement; the safe recommendation is schema authority only, with no destination deletion.
