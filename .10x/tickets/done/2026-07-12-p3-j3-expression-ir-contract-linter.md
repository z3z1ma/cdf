# P3 J3: expression IR and contract linter

## Scope

Define a stable CDF expression serialization that round-trips through the pinned DataFusion `Expr`; migrate declarative derive/filter/contract rules; use DataFusion simplification, function resolution, constant folding, and interval reasoning at plan time; lower recorded identity-bearing expressions exclusively to native fused kernels; permit `PhysicalExpr` only for nonidentity analysis/query work; export explicit-fidelity Substrait views.

## Acceptance criteria

- Derive/filter/contract use one semantic compiled expression plan and versioned serialization.
- Optimized expression, functions/versions, fidelity, residuals, and lints are recorded before execution.
- Replay never silently re-optimizes a recorded expression.
- Native fused lowering is the only identity path; unsupported native functions fail planning, while nonidentity `PhysicalExpr` use has differential Arrow/null/error tests.
- Linter detects provably unsatisfiable ranges, always-true rules, and cursor-subsumed filters without inventing conclusions under uncertainty.
- Kernel/contract public APIs expose no DataFusion Rust types.

## Evidence expectations

Parser/serialization goldens, native/DataFusion differential property tests, optimizer-version replay fixtures, function catalog coverage, Substrait fidelity cases, throughput comparison, and review.

## Explicit exclusions

No arbitrary SQL execution inside contracts and no runtime optimizer authority.

## Blockers

V2 established the fused validation graph seam; no remaining blocker.

## Review

- 2026-07-12 independent adversarial review (fresh reviewer; source inspection only, no build/test rerun).

### Findings

- **critical** — Recorded contract expressions are not execution authority. Planning creates `EnginePlan.contract_expressions` from the row rules (`crates/cdf-engine/src/planning.rs:185-196`), but no execution or preflight path reads or compares that field; package execution instead clones and evaluates `plan.validation_program` directly (`crates/cdf-engine/src/execution.rs:1660-1687`). The production backfill path also replaces `engine_plan.validation_program` after planning without rebuilding `contract_expressions` (`crates/cdf-cli/src/backfill_command.rs:125-133`). Therefore a changed/stale row-rule program can supply identity-bearing verdicts while the supposedly recorded optimizer/function/fidelity envelope remains unrelated. Required repair: validate a one-to-one binding between every executable row rule and its recorded expression before any source contact, and eliminate/replan post-planning program mutation.
- **significant** — Optimizer/function/fidelity compatibility is recorded but not enforced or persisted as package/replay authority. `PlannedExpression::validate_recorded` accepts any non-empty optimizer name/version and checks only Substrait fidelity equality (`crates/cdf-contract/src/expression.rs:250-264`); it does not reject a different DataFusion pin, function namespace/version set, IR version/dependency mismatch, altered optimized tree, Substrait version/expression/residual mismatch, or `Unsupported`/residual-inconsistent execution. Package creation writes scan, explain, graph, and validation-program artifacts but none of `predicate_expressions`, `residual_expressions`, or `contract_expressions` (`crates/cdf-engine/src/execution.rs:1673-1683`). The residual source-text count guard at `crates/cdf-engine/src/execution.rs:3070-3083` does not close those version and package gaps. Required repair: define and enforce the current compatible optimizer/function/fidelity tuple at plan consumption, persist the complete recorded compiled expression plan in the package, and add stale/tampered plan/package rejection fixtures.
- **significant** — Identity planning does not prove that the native evaluator supports the schema DataFusion accepted. The planner marks every successfully converted expression `Exact` (`crates/cdf-engine/src/expression.rs:20-45`) without a schema/type capability check against the native lowering. The native residual evaluator handles only Int32/Int64, UInt32/UInt64, Utf8, and Boolean (`crates/cdf-engine/src/predicates.rs:180-210`), omitting DataFusion-admitted/common Arrow types such as Int8/Int16, UInt8/UInt16, and LargeUtf8 and all decimal/temporal types. Such expressions can pass analysis and fail only after payload execution instead of failing identity-bearing planning. Required repair: bind native lowering against the planned Arrow schema (including casts/NULL semantics) and differentially test every admitted operator/type against DataFusion/Arrow semantics.
- **significant** — Declarative derive/filter expressions were changed to the CDF IR, but are absent from planning authority. `finish_plan` plans only scan request filters and row rules (`crates/cdf-engine/src/planning.rs:153-196`); `TransformDescription::{Derive,Filter}` expressions embedded in `ValidationProgram.transforms` receive no optimized envelope, function resolution, fidelity, lint, native capability check, or recorded residual. This does not meet the acceptance criterion that derive/filter/contract share the planned semantic authority.
- **significant** — The exported object is not a Substrait expression. `SubstraitExpressionView` contains the CDF `ExpressionNode` plus a string version/fidelity/residual list (`crates/cdf-contract/src/expression.rs:274-280`), and construction simply copies the optimized CDF node under `"substrait-expression-view-v1"` (`crates/cdf-engine/src/expression.rs:72-77`). It has no Substrait protobuf/JSON relation, extension/function anchors, type declaration, or round-trip interoperability proof. Rename it as a CDF interchange projection or implement an actual Substrait representation with explicit unsupported/lossy cases before claiming Substrait export.
- **significant** — The evidence does not test semantic parity. The sole round-trip test asserts one Int64 expression remains syntactically equal and that two metadata fields match (`crates/cdf-engine/src/expression.rs:541-555`); there is no native-vs-DataFusion evaluation comparison across NULLs, casts, types, errors, simplifications, or functions, no derive/contract parity matrix, and no optimizer-version/package/replay mismatch fixture. This leaves the ticket's differential, replay, and package acceptance criteria unsupported even if the implementation findings above are repaired.
- **minor** — The contract range linter parses bounds as i64/u64 for every signed/unsigned width and compares Utf8 bounds even though the vector `in_range` lowering admits numeric types only. Width-aware binding should be shared with the native range binder so lint conclusions and planning failures cannot drift.
- **nit** — `unsupported_identity_function_fails_at_plan_time` currently asserts the incidental `"unsupported arity"` text for a unary unknown function rather than an exact unsupported-function capability diagnostic.

### Verdict

**fail** — The CDF-owned versioned IR and crate layering are directionally correct, and the legacy `ParsedPredicate` type is removed, but identity-bearing contract execution is not bound to recorded compiled expression plan; optimizer/function/fidelity compatibility is not enforced or packaged; derive/filter planning, honest Substrait export, and semantic differential evidence are incomplete.

### Residual risk

Even after the critical binding repair, replay stability remains unproven until a persisted expression artifact is tamper/version tested and native/DataFusion parity covers the complete admitted schema/operator matrix. No claim is made about compilation of the final replay/lint edits because this review was explicitly source-only.

## Retrospective

- What worked: putting serialization in `cdf-contract` and every DataFusion conversion in one private engine module made the forbidden dependency boundary mechanically obvious. Deleting `ParsedPredicate` immediately prevented a tempting dual-semantics fallback.
- Surprise: contract rules already had a mature native vector/scalar lowering enum, so a shallow IR wrapper would have created two authorities. Replacing the serialized enum while deriving the same native bindings from IR was the smallest complete migration, but it widened the compile surface beyond scan filters.
- Dead end: the first implementation recorded only residual filters. That could have let legacy deserialized plans silently skip residual execution because the new field has a serde default. The final replay guard turns missing/mismatched authority into a planning error instead.
- Five-whys lesson: expression changes become replay changes because raw request strings coexist with compiled plans; serde compatibility can therefore weaken execution unless absence is explicitly rejected at the execution boundary. Future versioned-plan additions should pair tolerant deserialization with a mandatory runtime authority check.
- Five-whys addition: ordinal safety is not an integer-index property; it requires the recorded physical schema, name/type verification, and provenance-mask composition across every row-removing stage. The final fixtures now encode that invariant directly.
- Naming lesson: `Authority` obscured a concrete compiled artifact. A mandatory `CompiledExpressionPlan` communicates lifecycle and exclusivity without a governance metaphor; the execution-authority invariant belongs in documentation, not the type name.
- Follow-up ownership: broader function admission and real Substrait interchange are optional future scope under the parent bridge program. The admitted J3 surface has differential, replay, throughput, and adversarial closure evidence.
