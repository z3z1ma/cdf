# P3 J0: typed statistics evidence spine

## Scope

Replace the unpopulated lexical `BatchStats` and aggregate-only profile artifact with the CDF-native typed statistic model, vectorized batch computation, deterministic segment/package aggregation, opt-in canonical `stats/profile.parquet`, and verified streaming readers required by J1.

## Acceptance criteria

- Supported primitive/decimal/temporal/string/binary columns produce exact typed min/max/null/row evidence; unsupported or ambiguous cases explicitly remain incomplete.
- Computation is vectorized, memory-accounted, segment-streaming, jobs-invariant, and measured against the P3 overhead budget.
- When emitted, `stats/profile.parquet` is deterministic, manifest-bound, corruption-checked, and readable without payload access; disabled profile emission is a conservative absence for pruning, not a hot-path default.
- Lexical fields and aggregate `profile.json` are deleted with no dual representation or compatibility shim.
- Kernel/package APIs remain DataFusion-free and adding a destination/source/format requires no statistics-specific branch.

## Evidence expectations

Generated Arrow type/density arrays, scalar round-trip/property tests, jobs/golden packages, corruption/schema-generation adversaries, large-package RSS, primitive/wide/nested benchmarks, strict dependency checks, and adversarial review.

## Explicit exclusions

No predicate evaluation, DataFusion type, pruning decision, anomaly policy, or exact unbounded distinct set.

## Blockers

None. A5/V2 provide the fused streaming stage.

## Review

Fresh independent adversarial review of the first uncommitted milestone (2026-07-12):

### Findings

- **significant — The shared-memory lease does not own the actual retained statistics lifetime.** `persist_canonical_segments` reserves `2 * retained_bytes` before computing a segment, but package aggregation retains extrema accumulated across segments while `register_ready` keeps only the maximum single-segment lease. A sparse/wide adversary can place a large value in a different column in each segment, causing the package aggregate to retain the sum of those per-column extrema while the ledger records only the largest segment reservation. Field paths and canonical nested type/metadata allocations are also not included in `retained_bytes`. In addition, aggregation mutates `profile.statistics` before transferring accounting; when a package lease already exists, the code drops the segment lease before attempting to grow the package lease, after the segment has been registered and the durable-segment hook has run. This violates reserve-before-retain and admits a late memory error after downstream staged work. Finally, the public default `execute_to_package*` APIs supply no `ExecutionServices`, so `state.memory` is `None` and all statistics allocations are deliberately unaccounted on those package-producing paths. The successful jobs test proves release to zero only for the injected-host path; it does not prove reservation adequacy, pressure failure, or the default path. The milestone cannot claim memory-accounted computation until a lease sized from the retained aggregate (and schema/path overhead) is acquired/grown before mutation on every production path, with focused pressure/error/cleanup evidence.
- **significant — Removing `stats/profile.json` is not yet a coherent repository identity boundary.** The writer is gone and no source compatibility writer competes with the typed model, but the active conformance goldens `crates/cdf-conformance/golden/live-local-file-{v1,parquet-v1,postgres-v1}/expected.json` still require `stats/profile.json`, while `stats/profile.parquet` does not exist. The ticket explicitly acknowledges this, so it is honest that full J0 remains active, but the current tree intentionally disagrees with its identity-bearing golden authority and should not be treated as a completed/committable milestone before the canonical Parquet artifact and golden promotion land together.
- **minor — The recorded jobs-invariance test overstates its ordering evidence.** `operator_graph_compiles_from_capabilities_without_driver_name_dispatch` compares jobs 1 and 4, but its two three-row sample batches fit in one canonical segment. It exercises parallel partition opening and equality of one aggregate, not out-of-order multi-segment encode completion or order-independent checked aggregation. The canonical `next_registration` frontier makes the implementation plausible, but a multi-segment adversary is still needed before claiming the test proves canonical registration-order aggregation.

### Confirmed boundaries

- No DataFusion dependency or type appears in `cdf-kernel` or `cdf-package`; the new kernel dependency is pinned Arrow `arrow-arith` only.
- Supported signed/unsigned integer, boolean, decimal32/64/128, date/time/timestamp/duration, UTF-8, and binary branches use typed kernels/scans and retain the canonical Arrow declaration. Fresh final adversarial re-review after the boxed-ownership/pre-encode repair (2026-07-12; requested 5.6-sol/high was unavailable in the collaboration API, so this is a fresh review agent):

### Final re-review findings

- **significant — Nested metadata is capacity-exact but not canonical when constructed from Arrow, so logically equal schemas can drift or fail round-trip equality.** `StatisticsArrowField::from_arrow` collects `Field::metadata()` directly into the boxed slice without sorting it. Arrow 59 stores field metadata in a `HashMap`, whose iteration order is not semantic or deterministic. Two independently constructed but Arrow-equal nested fields can therefore produce differently ordered `StatisticsArrowType` values and make `merge_owned` reject them as a type change; serialization also emits the unsorted order, while `boxed_metadata::deserialize` sorts it, so a typed value with two or more nested metadata entries is not guaranteed to equal its own serde round trip. The repaired representation correctly avoids spare capacity and allocator-opaque map nodes, and `retained_bytes` recursively charges the boxed entry array and both payloads, but the API/serde semantics still violate the claimed sorted canonical declaration. In the same pre-computation authority, `arrow_type_description_bytes` omits the two boxed `StatisticsArrowType` allocations for every `Dictionary` node even though `canonical_type_heap_bytes` correctly charges them after construction; the `2 * schema_bytes` term therefore is not itself the claimed structural bound and currently relies on unrelated input-array allowance to cover the omission.
- **significant — Moving statistics growth ahead of its own encode does not eliminate unowned files from already-active encoders when a later segment fails.** With more than one encode in flight, segment N can already be inside `PackageSegmentEncoder::encode` while segment N+1 computes statistics and fails reservation, checked merge, or reconciliation before its own submission. Dropping `SegmentEncodeQueue` cancels and joins workers but neither drains/registers successful completions nor deletes their published IPC files; `EncodedPackageSegment` has no cleanup-on-drop behavior. The current segment now correctly fails before its own publication, and successful completions still register/hook in canonical manifest-last order, but the package-wide failure path can leave an earlier worker's hash-complete file outside both artifact-receipt and segment-draft journals. A focused concurrent late-statistics-failure adversary plus file/journal cleanup assertion remains required before claiming the A5 rollback invariant.
- **minor — The jobs adversary permits two encoder workers only on hosts exposing at least three logical CPU slots and still does not demonstrate reordering.** The 512 MiB budget raises memory parallelism above one, but `maximum_in_flight` is also capped by `logical_cpu_slots - 1`; on one-core hosts the queue is inline and on two-core hosts it is one-wide, while the test merely asserts output equality and ledger release. Even on a larger host, one-row IPC encodes may complete before the next submit and the test records neither peak in-flight count nor a reversed completion order. Thus it is a useful six-segment jobs-1/jobs-4 identity law, but not yet portable evidence that the reorder frontier was exercised.

## Retrospective

- Reusing `CanonicalArrowType` avoided a second type-declaration vocabulary and kept the kernel DataFusion-free.
- Computing only after canonical batch shaping makes the jobs-invariance argument structural: segment membership and order already belong to A5, and statistics no longer depend on source batch scheduling.
- The old aggregate JSON artifact cannot safely evolve into the typed contract, but deleting it before canonical Parquet lands creates an incoherent identity boundary. Keep it byte-compatible and exclude typed in-memory evidence from its serializer until writer, reader, verification, corruption checks, and golden promotion replace it together.
- Rust generic abstraction over Arrow string/binary arrays increased compile complexity without reuse value. Concrete array branches are clearer and match the type-specialized hot-path requirement.
- No follow-up ticket was opened for the remaining artifact/read/performance work because this active J0 ticket is already its exact durable owner; duplicating it would split authority. The next executor should define the one versioned Parquet row schema, stream canonical segment rows as registration occurs, derive bounded package/file rows, bind and verify them against the finalized manifest, then promote package goldens and run corruption/jobs/RSS/overhead evidence.
