# P3 C2 scheduler-owned source retry boundary

## Observation

CDF now compiles one typed retry policy from source execution capabilities and the identity proof carried by each exact planned partition. A source-wide capability therefore cannot authorize retry for a weakly identified HTTP or local object. The runtime scheduler owns attempt count, elapsed deadline, full-jitter delay, cancellation, typed eligibility, retry history, and successful outcome admission. HTTP and REST no longer own independent attempt budgets.

The implemented execution boundary retries a partition open or its first lazy stream poll before any batch becomes visible to canonical processing. Every invocation-bound stream carries an awaitable producer-termination barrier; the scheduler joins it before reopening, accepting a partial limit, or returning a cancellation/error. A successful retry is reattested before reopen and after consumption: through EOF completion for a full scan or an explicit post-consumption probe for a partial limit. A changed generation/schema aborts the run and leaves no finalized package. Mid-stream retry remains fail-closed after the first visible batch pending exact resume or attempt-local staging.

Retry history is validated execution evidence and is also appended to the non-package run ledger. The scheduler journals each observed decision before awaiting its delay, so cancellation cannot erase an attempt that already happened. A runtime journal survives an engine error return, so attempt exhaustion is not lost merely because no successful package exists. Timing and jitter remain outside package identity.

The engine validates the live canonical schedule against both its explained schedule and an independently compiler-bound source-semantics/execution-capabilities identity before source contact. This rejects coherent execution-ceiling plus schedule widening, not only schedule-only tampering. On every orchestrated engine exit, opened streams are terminated and joined, including speculative streams already opened behind a failed canonical head. HTTP rate limiting is classified as `RateLimited`, and integer-seconds `Retry-After` reaches the single scheduler policy rather than creating a transport-local attempt loop.

The invocation lifecycle is source-neutral. `InvocationTermination` carries both cancellation and an eager awaitable join. Files, REST, Postgres, Python, declarative SQL, and the DataFusion table adapter retain this handle through opening, stream failure, EOF, limit termination, and cancellation. `PartitionOpenAttempt` alone transfers that handle into an opened stream; an adapter payload cannot substitute another invocation's handle. Declarative SQL supplies a connection resolver to the Postgres source adapter, which resolves the secret and performs database work inside one source-owned blocking invocation rather than nesting facade and adapter lifecycles. Retry journals accept only prefix-monotonic history for the exact scheduled-partition binding, and durable run events persist the complete compiled policy and binding hash.

## What it supports

- One scheduler-level attempt/deadline/backoff authority exists and adapter-local HTTP/REST retry machinery is deleted.
- The ratified default is executable through injected monotonic-clock, entropy, and cancellation services without putting timing or entropy in package identity.
- Unit-granularity policy cannot widen into the partition-open retry boundary, and per-partition retry safety prevents a source-wide maximum capability from blessing weak file identities.
- A retry starts only after the failed `open` future terminates, or after an invocation-bound stream reports its first terminal error and its mandatory producer barrier joins; no hedged or duplicate attempt exists.
- Retry history is validated, partition-ordered runtime evidence; it survives failed engine returns in the durable run ledger, and only one successful stream crosses the canonical frontier.
- Retry history is append-only by exact prefix for one scheduled-partition binding; durable events retain the complete compiled retry policy and cannot be interpreted through a mutable display id alone.
- Retry decisions enter the journal before backoff, terminal ineligible evidence cannot replace the primary typed error, and a tampered live retry schedule fails before source contact.
- Every engine/DataFusion-opened invocation is terminated and joined on success, error, cancellation, partial-limit completion, or speculative-frontier cleanup; reopening cannot overlap a live prior attempt.
- Exact file generation revalidation is one metadata read per selected object after one planning enumeration. Object-store execution does not relist the glob, and HTTP numeric-template membership does not allocate or scan the complete expansion per partition.
- Zero-work plans are true no-ops at both engine and DataFusion execution boundaries; neither path opens a source to prove that it needs no rows.
- Completion or partial-limit reattestation rejects a changed source generation/schema before package finalization while permitting terminal extraction to strengthen metadata-only file evidence with the content hash it just computed.
- Structured scope cancellation no longer races with the primary error or leaves retry timer work detached.
- Incremental file-manifest planning cannot sanitize or skip a tampered compiled schedule/source ceiling: project preflight validates both authorities before state lookup, partition subsetting, and the no-change fast return.
- Project execution has one services-mandatory production path. Staged destination ingress, engine execution, and replay memory all consume the caller's injected authority; no internal default host or fallback coordinator can silently replace it.

## Limits

- This is the atomic open/pre-first-batch boundary, not general mid-stream retry. A stream error after a visible batch remains terminal until exact resume or attempt-local staging exists.
- REST and Postgres advertise no scheduler retry because they do not yet provide immutable/snapshot attestation.
- This evidence does not close C2's randomized jobs-invariance matrix, fair non-file admission, limited speculation, or scale profile acceptance criteria.
- Exact limited-run slicing remains open: the current consumer may poll once beyond the selected row prefix to distinguish a boundary-aligned limit from EOF. It does not advance or persist the discarded position, but eliminating that source observation belongs to the remaining one-step frontier/limit design and is not claimed here.
- The focused suite and workspace compile do not substitute for slow stress, chaos, or release-envelope execution.
- A broad `cdf-conformance` run is not closure evidence for this slice: it exposed existing program-level golden/catalog/replay/checksum failures outside the focused retry and resource-execution laws. C2 remains active, and those failures must stay owned rather than being hidden by a green focused result.
- Run-ledger retry events are appended after the engine returns. They survive ordinary terminal failures but not an abrupt process kill in the middle of an attempt; crash-durable per-attempt telemetry would require an injected event sink and is not claimed here.
- The pre-existing `cdf-project` processed-observation failure remains outside this evidence; the clean-HEAD comparison proves only that this retry slice did not introduce it, not that the integration layer is healthy.
