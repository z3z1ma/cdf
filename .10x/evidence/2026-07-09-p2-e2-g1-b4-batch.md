# P2 E2/G1/B4 batch verification

## What was observed

The E2, G1, and B4 children are implemented and verified together:

- E2: HTTPS single-file Parquet resources now use the file transport facade for bounded discovery and production plan/preview/run. HTTP file metadata/range/read calls flow through the CLI's reqwest-backed transport, with auth and egress denial before client use.
- G1: `cdf validate --deep` is a real CLI doorway and source diagnostics now include compiled ids, origins, project mapping status, likely causes, suggestions, `CDF-RESOURCE-NOT-COMPILED`, and command-correct wording for the covered plan/validate surfaces.
- B4: `cdf-conformance` owns property tests for widening lattice composition and no-loss materialization across signed integers, unsigned integers, float32-to-float64, and date32-to-timestamp.

Golden package hashes changed only because file partition metadata now includes the `sha256` identity field in live-run plan evidence. The affected live-run expected JSON fixtures were updated after conformance showed the new deterministic package identities.

## Limits

This evidence does not close P2. E2 is scoped to deterministic HTTPS single-file Parquet, not HTTP template enumeration, cloud object stores, recursive remote globs, compression, or the public live TLC S1/S2 session. G1 is the diagnostics/deep-validate foundation, not the final cloud/compression/Python/WASM deep compiler matrix. B4 covers widening properties for the implemented no-loss Arrow families, not every future cast or validation-program serialization path.
